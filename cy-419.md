# cy-419 代码迷城 DebugQuest

请根据以下要求生成项目代码和 README。

## 项目主要需求

### 1. 项目概述

**代码迷城 DebugQuest** 是一款基于 Canvas 的浏览器端 Roguelike 解谜游戏。玩家扮演一名 "Bug Hunter"，在由代码片段构成的地下城中探索，收集语法碎片、修复运行时 Bug、击败异常怪物，最终抵达 "Production" 层。每一层地图随机生成，Bug 类型对应不同异常（NullPointer、StackOverflow、TypeError 等），修复 Bug 可解锁新区域。项目类型：**浏览器端游戏开发**。

### 2. 核心数据模型（≥ 3 个，必须独立成文件）

- **Player（玩家角色）**
  - 字段：id, name, hp, max_hp, mp, max_mp, level, exp, attack, defense, position（x, y）, inventory（道具数组）
  - 关键关系：玩家在每个 Dungeon 层中移动，与 Bug、Item 交互
  - 业务动作：移动、攻击、拾取道具、使用技能、升级
  - **此模型必须独立定义在 `src/models/player.ts`**

- **Dungeon（地牢层）**
  - 字段：id, level, width, height, tiles（二维数组：地板/墙/门/陷阱）, fog_map（迷雾）
  - 关键关系：包含多个 Bug 和 Item
  - 业务动作：生成地图、渲染迷雾、切换楼层
  - **此模型必须独立定义在 `src/models/dungeon.ts`**

- **Bug（异常怪物）**
  - 字段：id, type（null_pointer / stack_overflow / type_error / memory_leak / syntax_error）, hp, attack, defense, position, status（alive/dead）, drop_item_id
  - 关键关系：属于某个 Dungeon，与 Player 战斗
  - 业务动作：移动、攻击玩家、死亡掉落
  - **此模型必须独立定义在 `src/models/bug.ts`**

- **Item（道具/语法碎片）**
  - 字段：id, name, type（potion / syntax_fragment / debugger / refactor_scroll）, effect_value, position, rarity
  - 关键关系：散落在 Dungeon 中，被 Player 拾取
  - 业务动作：拾取、使用、加入背包
  - **此模型必须独立定义在 `src/models/item.ts`**

### 3. 核心页面/场景（≥ 4 个，至少 2 个页面共用同一业务组件）

- **/start（开始界面）**
  - 消费模型：Player
  - 关键交互：输入玩家名、选择难度、开始游戏、查看排行榜
  - 共享组件：`<GameButton>`、`<LeaderboardPanel>`

- **/game（游戏主场景）**
  - 消费模型：Player、Dungeon、Bug、Item
  - 关键交互：键盘移动、战斗、拾取、切换楼层、暂停
  - 共享组件：`<GameCanvas>`、`<StatusBar>`、`<MiniMap>`、`<CombatLog>`

- **/leaderboard（排行榜）**
  - 消费模型：Player（历史最高分）
  - 关键交互：查看排名、返回开始页
  - 共享组件：`<LeaderboardPanel>`（与 /start 共用）、`<GameButton>`（与 /start 共用）

- **/settings（设置）**
  - 消费模型：无
  - 关键交互：调整音量、切换主题、重置存档
  - 共享组件：`<GameButton>`（与 /start 共用）

- **/game-over（结算界面）**
  - 消费模型：Player
  - 关键交互：显示本局成绩、保存到排行榜、重新开始
  - 共享组件：`<GameButton>`、`<StatusBar>`（与 /game 共用）

### 4. 横切关注点（≥ 2 个）

- **全局状态管理（游戏状态机）**
  - 触达文件层：
    - `stores/gameStore.ts`
    - `stores/playerStore.ts`
    - `stores/dungeonStore.ts`
    - 多个场景组件读取状态

- **游戏日志系统**
  - 触达文件层：
    - `utils/gameLogger.ts`
    - `constants/logTemplates.ts`
    - `components/common/CombatLog.tsx`
    - 所有战斗、拾取、升级事件都记录日志

- **全局错误处理/存档恢复**
  - 触达文件层：
    - `utils/saveManager.ts`
    - `components/common/GlobalErrorBoundary.tsx`
    - 所有关键操作后自动存档

### 5. 共享枚举/常量（≥ 2 组）

- **BugType（异常类型）**
  - 值：NULL_POINTER = 'null_pointer', STACK_OVERFLOW = 'stack_overflow', TYPE_ERROR = 'type_error', MEMORY_LEAK = 'memory_leak', SYNTAX_ERROR = 'syntax_error'
  - 定义位置：`src/constants/bug.ts`
  - 出现位置：Bug 模型、怪物生成器、战斗计算、日志模板、状态文本、音效配置、粒子特效
  - **README 必须列出该枚举的所有出现位置**

- **ItemType（道具类型）**
  - 值：POTION = 'potion', SYNTAX_FRAGMENT = 'syntax_fragment', DEBUGGER = 'debugger', REFACTOR_SCROLL = 'refactor_scroll'
  - 定义位置：`src/constants/item.ts`
  - 出现位置：Item 模型、道具生成器、背包 UI、使用效果、日志模板、formatters
  - **README 必须列出该枚举的所有出现位置**

### 6. 共享前端组件清单（≥ 3 个）

- `<GameCanvas>`：被 /game 使用
- `<StatusBar>`：被 /game 和 /game-over 共用
- `<MiniMap>`：被 /game 使用
- `<CombatLog>`：被 /game 使用
- `<GameButton>`：被 /start、/leaderboard、/settings、/game-over 共用
- `<LeaderboardPanel>`：被 /start 和 /leaderboard 共用
- `<InventoryPanel>`：被 /game 使用
- `<EmptyState>`：被排行榜空状态使用

### 7. 共享 hooks/utils 清单（≥ 2 个）

- `useGameLoop()`：游戏主循环管理
- `useKeyboard()`：键盘事件监听
- `useCombat()`：战斗逻辑封装
- `utils/gameLogger.ts`：游戏日志记录
- `utils/saveManager.ts`：localStorage 存档读写
- `utils/randomDungeon.ts`：随机地图生成
- `utils/combatCalculator.ts`：伤害计算、经验计算
- `utils/formatters.ts`：Bug 类型文本、道具类型文本、数值格式化

```

├── gameStore.ts
├── playerStore.ts
├── dungeonStore.ts
└── settingsStore.ts
```

禁止全部用组件内部 state。每个 store 按职责分文件，组件中按需引入。

### 10. 文件结构强制清单

```
src/
├── game/              # Phaser 游戏场景与逻辑
│   ├── scenes/        # BootScene, StartScene, GameScene, GameOverScene
│   ├── entities/      # PlayerEntity, BugEntity, ItemEntity
```

**严禁合并职责到单一文件。即使是简单功能，也必须按照上述分层结构拆分，确保后续每一个小改动都至少触达 3-5 个文件。**

### 11. 特殊约束：屎山代码设计（低内聚、高耦合、牵一发动全身）

1. **日志模块必须单独管理但全游戏引用**
   - 必须有一个 `utils/gameLogger.ts`
   - 所有游戏系统、组件、store 都引用 logger
   - 日志模板集中在 `constants/logTemplates.ts`，包含至少 30 条模板
   - 任何新 Bug 类型或道具类型都必须同步新增日志模板并在调用处使用

2. **异常信息/提示文案必须分散**
   - `constants/messages.ts` 同时包含游戏提示、战斗提示、UI 文案、日志文案
   - 多个组件、utils、store 都直接引用 messages
   - 修改一个文案可能同时影响 UI、日志和提示

3. **常量/工具类必须多处耦合**
   - `utils/formatters.ts` 同时包含 Bug 类型文本、道具类型文本、数值格式化、稀有度文本、楼层文本
   - `constants/dungeon.ts` 同时包含楼层配置、地图尺寸、怪物生成概率、陷阱伤害
   - 修改一个数值可能同时影响生成器、战斗、UI 显示

4. **动画/音效配置必须跨多处**
   - Bug 出现动画、死亡动画、音效 key 在 `constants/animations.ts`、entities、scenes、utils 中同时存在
   - 新增 Bug 类型需要同步配置动画和音效

5. **枚举必须被多处引用**
   - BugType 和 ItemType 必须在 models、constants、game systems、components、stores、formatters、logTemplates、animations 中同时存在
   - 新增一个枚举值时，必须修改至少 10 处文件

6. **牵一发动全身的验证标准**
   - 后续如果要求"新增一种 Bug 类型 'race_condition'"，必须触达：Bug 模型、BugType 枚举、随机生成器、战斗系统、formatters、messages、logTemplates、animations、音效配置、粒子特效、状态栏、CombatLog、GameOver 统计等 ≥ 12 个文件
